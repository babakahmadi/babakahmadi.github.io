Solid Principles

As a programmer, I reviewed a lot of codes(specially in java) and I saw a lot of problems in them. One of the most common problems that I saw is violating the SOLID principles. You know, when I started programming with java, I had these kind of problems too and I didn’t know these principles but after knowing that, my codes were better than before and I think these kind of principles prevents us from very large number of issues in programming and help us to scale our code, so it would be great to write a blog about it. Here, I try to define these principles and tell some examples that would be good for better understanding (all my examples will be in java).

hint: I used the lombok annotations, to know about that you can check this lombok

SOLID is acronym of five principles in object oriented design which is declared by uncle Bob. SOLID principles stands for:

  • S ingle-Responsiblity
  • O pen-Closed
  • L iskov Substitution
  • I nterface Segregation
  • D ependency Inversion

Before every step, I think it’s good to show the first implementation of Order and Payment for our basic and naive service.

@Slf4j
public class Order{
    List<String> items;
    List<Integer> quantities;
    List<Integer> prices;
    String status;

    public void addItem(name, quantity, price){
        this.items.add(name);
        this.quantities.add(quantity);
        this.prices.add(price);
    }

    public int calculatePrice(){
        int total = 0;
        for(int i=0;i<this.items.size();i++){
            total += this.quantities.get(i) * this.prices.get(i);
        }
        return total;
    }

    public void pay(String paymentType, int amount, String securityCode){
        switch(paymentType){
            case "debit":
                log.info("Processing with debit for amount {}", amount);
                this.status = "paid";
                break;
            case "credit":
                log.info("Processing with credit for amount {}", amount);
                this.status = "paid";
                break;
            default:
                throw new RunTimeException("unknown payment method");
        }
    }
}

Here we have an Order class that has some attributes about the order: items, quantities, prices and status. This class has some methods to add item, calculate price and pay the order.So, after introducing our base code, lets go the the main topic.

Single responsibility

A class should do one thing and therefore it should have only a single reason to change. lets take a look back on our first implementation. The Order class is something about order, so addItem and calculatePrice are its responsibility but as it’s clear the pay method is not responsibilty of the Order. why? because it’s about payment. It’s very obvious that this functionality needs to be extracted from the order and goes to another class.

@Slf4j
public class PaymentService{
    public void payDebit(int amount, Order order, String securityCode){
        log.info("Processing with debit for amount {}  with security code {}", amount, securityCode);
        order.setStatus("paid");
    }
    public void payCredit(int amount, Order order, String securityCode){
        log.info("Processing with credit for amount {} with securityCode {}", amount, securityCode);
        order.setStatus("paid");
    }
}

Here, we create a PaymentService class that is responsible for the pay method and we separated pay method to two methods named payCredit and payDebit. so we resolved the problem with Single Responsiblity. It helps us to reuse these classes much easier in our service. The other thing is that if we want to add some other payment methods later, we don’t need to add these methods to the order class.

Open-Closed

classes should be open for extension and closed to modification. Lets take a look to the PaymentService class. If we want to add a new payment method to the PaymentService we need to modify the class. This is the main reason to disrupt the Open-Closed principle. To resolve it we need to extract each payment method as a class and to add new payment method we can create the new one.

public abstract class PaymentService{
    public void pay(int amount, Order order, String securityCode);
}

@Slf4j
public class DebitPaymentService extends PaymentService{
    @Override
    public void pay(int amount, Order order, String securityCode){
        log.info("Processing with debit for amount {}  with security code {}", amount, securityCode);
        order.setStatus("paid");
    }

}

@Slf4j
public class CreditPaymentService extends PaymentService{
    @Override
    public void pay(int amount, Order order, String securityCode){
        log.info("Processing with credit for amount {} with securityCode {}", amount, securityCode);
        order.setStatus("paid");
    }
}

With this change, we don’t violate the Open-Closed principle, because if we need to add new payment method like paypal, we can add new class that inherits the PaymentService and we don’t need to modify the classes (and we can extend the the classes). Here you are:

@Slf4j
public class PaypalPaymentService extends PaymentService{
    @Override
    public void pay(int amount, Order order, String securityCode){
        log.info("Processing with paypal for amount {} with securityCode {}", amount, securityCode);
        order.setStatus("paid");
    }
}

Liskov-Substitution

subclasses should be substitutable for their base classes. I always call this principle Lifskov, Lifisiovski, Loftuschik or something like this and I know I will forget this name in near future (about an hour). So what does it mean here? Lets take a look. The paypal method is not using the securityCode and instead of that, it uses the email. The first idea is that we can change the pay method attributes to this:

public void pay(int amount, Order order, String email){
    log.info("Processing with paypal for amount {} with email {}", amount, email);
    order.setStatus("paid");
}

but it has a semantical problem with the parent class, because the pay method uses securityCode and here we used the email. This is the reason for violating the Liskov-Substitution. To fix it, we can extract the securityCode and email from the method and add it as a field and set it with the Constructor. Here’s the new code:

public abstract class PaymentService{
    public void pay(int amount, Order order);
}

@Slf4j
public class DebitPaymentService extends PaymentService{
    private String securityCode;
    public DebitPaymentService(String securityCode){
        this.securityCode = securityCode;
    }

    @Override
    public void pay(int amount, Order order){
        log.info("Processing with debit for amount {}  with security code {}", amount, this.securityCode);
        order.setStatus("paid");
    }

}

@Slf4j
public class CreditPaymentService extends PaymentService{
    private String securityCode;
    public DebitPaymentService(String securityCode){
        this.securityCode = securityCode;
    }
    @Override
    public void pay(int amount, Order order){
        log.info("Processing with credit for amount {} with securityCode {}", amount, this.securityCode);
        order.setStatus("paid");
    }
}

@Slf4j
public class PaypalPaymentService extends PaymentService{
    private String email;
    public DebitPaymentService(String email){
        this.email = email;
    }
    @Override
    public void pay(int amount, Order order){
        log.info("Processing with paypal for amount {} with email {}", amount, this.email);
        order.setStatus("paid");
    }
}

now we substituted the method in child class based on the parent class. Lisokofpth-substitution resolved.
